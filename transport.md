# Shell, SSH, and API Reliability

Use these rules whenever a command crosses a shell, SSH, HTTP, JSON, or encoding boundary:

1. Identify every parser in the path before running the command, such as PowerShell -> native argv -> SSH -> remote shell -> JSON. Minimize the number of parsing layers instead of adding escapes until a command happens to work.
2. In PowerShell, invoke executables with `&` and an argument array. Prefer literal single-quoted strings for fixed text. Never use `Invoke-Expression`, and do not interpolate untrusted or secret values into command strings.
3. Never assume the machine's locale or default text encoding. Use UTF-8 explicitly for files, standard input, standard output, subprocesses, and network payloads. On Windows, enable Python UTF-8 mode with `PYTHONUTF8=1` or `python -X utf8` when a tool does not specify its own encoding; set PowerShell and console encodings deliberately when piping non-ASCII text.
4. Treat native-program exit codes as authoritative. PowerShell may surface native stderr as error records, while tools such as search, Git, and GitHub CLI may use nonzero codes for expected states such as no matches or a missing repository. Interpret each tool's documented result before classifying it as a failure.
5. Validate shell-specific syntax before combining operations. Do not assume Bash syntax works in PowerShell or vice versa. Keep commands fail-fast and check `$LASTEXITCODE` after important native operations so a later successful command cannot hide an earlier failure.
6. In PowerShell, wrap command or pipeline output in `@(...)` before relying on `.Count`, numeric indexing, or collection comparisons. A single result may otherwise collapse to a scalar, and indexing a string returns a character rather than a path or record.
7. PowerShell statement-style control constructs such as `foreach (...) { ... }` cannot be piped directly. Assign or collect their output first, or wrap the construct in `@(...)` or `$(...)`, before sending it through a pipeline.
8. For SSH, keep the remote command small and fixed. PowerShell OpenSSH commands containing spaces, JSON, pipes, regular expressions, or inline code must use a single-token argument where valid or an exact UTF-8 script encoded as base64 and decoded remotely; do not rely on nested quote escaping. Send secrets and non-ASCII files with a binary-safe file transfer. Base64 is transport encoding, not encryption; never treat it as secret protection.
9. Build JSON with a serializer, not manual quote escaping. Send request bodies from standard input or a temporary file with an API client's binary/file-body option. Use proper URL/query encoders rather than concatenating parameters.
10. Pass secrets through the provider's credential mechanism, protected files, environment variables, or standard input as appropriate. Do not place credentials, tokens, cookies, private keys, or decoded secret payloads in command history, logs, tracked files, or diagnostic output.
11. Before committing in a new worktree, verify repository-local Git identity and remotes. If identity is missing, derive a suitable repository-local identity from the authenticated Git host account when unambiguous; otherwise ask. Do not change global Git identity merely to make one commit succeed.
12. Treat Git-host CLI authentication, HTTPS credentials, and SSH keys as separate transports. Before an SSH push, verify that the current process can use an authorized key; a logged-in host CLI does not prove SSH access. If SSH is unavailable but the host CLI is authenticated, use its HTTPS credential helper rather than retrying SSH unchanged or exposing a token in a URL or command.
13. Capture the exit code, stderr, and HTTP status before retrying. Treat malformed commands, validation responses, authentication failures, missing tools, locale failures, and quoting/encoding errors as deterministic; fix the cause instead of retrying unchanged.
14. Retry only genuinely transient failures such as connection resets, timeouts, rate limits, and selected 5xx responses. Use bounded retries with backoff and honor `Retry-After`. Reads may be retried when safe; never blindly retry a write unless it is idempotent, uses an idempotency key, or remote state is checked first.
15. When a cross-boundary command fails, reduce it to the smallest read-only probe and validate each boundary independently: local serialization and encoding, executable availability, transport connectivity, remote interpreter, authentication, then the final operation.
16. Preserve bundled, vendored, and unrelated user files. If a check reports a pre-existing formatting issue in such a file, isolate and report it instead of silently rewriting upstream content or weakening checks for the files being changed.
17. Verify state after remote writes using a separate read. Report the exact layer that failed and preserve useful sanitized diagnostics instead of labeling every failure as transient.
